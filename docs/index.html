Introduction

  • Core Foundations : iObserve, CIPM-Pipeline
  • Author : David Monschein

This documentation explains step by step how the new approach for maintaining consistency can be set up and executed. Since the implementation is rather complex, we will focus on the main use cases. The explanations are based on the included running example. Mostly there are also screenshots attached, which illustrate the use in a graphical way.

This approach makes it easier to keep architecture models and source code consistent (including the runtime environment). The Palladio Component Model (PCM) is used as modeling approach for software components. The core is the pipeline, which extracts updates for architecture models from monitoring data at runtime. The only part of the consistency maintenance that is only partially covered by the approach is the repository model of the PCM. This was already considered in previous approaches (see EJBMox and SoMoX). Therefore, an already initialized repository model with components and interfaces is required to use our approach. For example, you can extract the repository model from the source code with the previously mentioned approaches EJBMox and SoMoX.

Setup Headless PCM #back to top

The simulations of the architectural models are an important part of the approach. In order to be able to perform these simulations in a high-performance and encapsulated manner, the PCM headless simulator is used (see PCM-Headless). There are two ways to set up the simulator:

  1. Setup using Docker (much easier) - see Setup using Docker
  2. Setup using Gradle - see Setup using Gradle

If the simulator has been set up successfully, its web UI can be accessed at http://127.0.0.1:8080/index.html (the port 8080 may differ, depending on the configuration). If you have any further questions, please consult the corresponding Wiki.

Setup Development Environment #back to top

The following steps are necessary to set up a development environment for the project:
  1. Installation of the Java Development Kit (JDK) version 8 or higher (version 13 is recommended).
  2. Installation of the Eclipse IDE, required features are Gradle and EMF integration. Therefore we recommend to use the "Eclipse IDE for Enterprise Java Developers" and the release "2020-03" (see Eclipse IDE for Enterprise Java Developers - Release 2020-03). You should also use the newest version of Gradle (currently this is 6.4.1) to build the project within Eclipse.
  3. Optional: In order to be able to open and edit all Ecore models, it is necessary to install the Palladio Component Model (PCM) as Eclipse plugin. See the following instructions: PCM Installation; version 4.1.0 has been used during development.
  4. Install Lombok into your Eclipse IDE by downloading the installer from the referenced page and point it to your Eclipse location. More detailed instructions can be found here.
  5. Clone the Git repository where the code is located (either Link 1 or Link 2 - these are synchronized).
  6. Import the Gradle projects by selecting the following menu item of Eclipse: File → Import and in the following list select "Existing Gradle project". In the following dialog, point to the directory “dmodel.root” of the cloned repository (step 5).
  7. Finish the import and wait until it finished. After that, all subprojects should be available and no build errors should have occurred.

Many parts of the implementation depend on the automated validation of the architectural models. The validation requires a running instance of the Headless-PCM Simulator (see previous section).

Starting Application#back to top

In Eclipse, create a new run configuration for a Java application. As main class select the class "TriggerDModelRuntime", located in the "dmodel.app.start" project in the package "dmodel.app.start". In addition, a path to the configuration must be passed via the command line, this can be done in the tab "Arguments". As program argument, the path to the configuration must be specified, e.g. --config="C:\...\defaultConfig.yml". The port that should be used for the web UI can be passed as VM argument. Here you should use a port that is different from 8080, because this port is usually used for the headless simulator. The port can be set as follows: -Dserver.port=8090.

The configuration just needs to have the correct syntax, an example configuration can be found in the root project under the file name "defaultConfig.yml". This configuration can be used as a starting point, the adjustments to the configuration can be done later via the Web UI. The following image shows graphically how the run configuration in Eclipse should look like.

After the successful start of the application, the web interface should be available at http://127.0.0.1:8090/ (the port may differ depending on the configuration). It is normal that the dashboard on the start page shows some errors, the next sections of this manual will explain the correct configuration in more detail.

Configuration#back to top

In this section we will take a closer look at how to configure the approach correctly. There are basically three parts of the configuration:

  1. The project configuration, i.e. which application should be analyzed with the approach
  2. The model configuration, i.e. the paths to the individual models used by this approach
  3. Miscellaneous configuration, this includes for example the configuration of the simulations (simulation time, ...)

A running example is integrated in the Github repository of the approach, which can be used to test the configuration. It is located in the project "dmodel.example.primes". At the end of this section there are also screenshots of how a correct configuration should look like.

Project configuration

In the project configuration, you select the Java project that should be examined. It is also important that you specfify the folders which contain the source code files. To configure this, first select "Configuration" in the sidebar on the left and then "Project configuration" (http://127.0.0.1:8090/config/projectprops.html). In the following view, the path of the Java application should be entered first. As already mentioned, the project "dmodel.example.primes" can be used for this. First, you enter the path to this project and then you select the folder src/main/java as source code folder (as shown in the picture at the end of the section). Thereafter this part of the configuration can be saved by clicking on "Save Configuration".

Models configuration

To configure the paths to the models, the submenu item "Models Paths" must be selected in the sidebar under the menu item "Configuration" (http://127.0.0.1:8090/config/modelprops.html). All eight model paths should then be filled in on the left side. The required models can be found in the running example project in the subfolder "models_runtime". After all paths have been entered, the configuration can be saved.
An important note: if you want to instrument the project later on, you should press the button "Extract Mapping from Source Code" on the right side.

Miscellaneous configuration

In this part of the configuration all remaining adjustments are done. By default, this part of the configuration is already error-free, unless the headless simulator is not accessible at the predefined URL. In other words, the configuration of these properties is only optional. However, if you want to adjust various properties, first select the "Conceptual properties" item in the sidebar under "Configuration" (http://127.0.0.1:8090/config/conceptual.html). The first section of the subsequent view allows you to configure the transactionality of the pipeline. In the second section you can customize the validation process (especially the simulations).

After all parts have been configured correctly, the configuration should now be reported as error-free in the dashboard under "Overview" (http://127.0.0.1:8090/index.html).

Instrumentation #back to top

The integrated instrumentation function can be used to transform the application to be monitored so that it generates monitoring data and sends it to the backend. This can be done using the Web UI. To do so, select the following menu item in the sidebar: "Design-Time → Instrumentation" (http://127.0.0.1:8090/design/instrumentation.html).

Three settings are required:

  1. The output path, i.e. where the instrumented project should be stored
  2. The URL where the monitoring is able to access the instrumentation model - usually the respective address is displayed directly in the form.
  3. Enable/ disable logarithmic scaling, in the default setting it should be deactivated

You can also specify whether the mapping should be extracted from comments in the code before the instrumentation. You should activate this if the mapping has not yet been established. After clicking on "Start Instrumentation" the instrumentation process is started and the progress can be observed on the screen.

Design-Time System Extraction #back to top

Our approach also supports the semi-automatic (user-assisted) extraction of a system model at design-time. This process requires an existing repository model. During the extraction process, conflicts may occur and must be resolved by the user. These conflicts are unavoidable because certain information is not available at design-time (or only the developers know it). However, the number of conflicts can be reduced by first building a service call graph (SCG) from the code. This graph stores information about which service calls which other services. The SCG enables the automatic resolution of some conflicts. The process of building a SCG is based on a code analysis. Therefore the number of conflicts strongly depends on the quality of the code analysis.

Service-Call-Graph (SCG) Extraction

In order to automatically build the Service Call Graph (SCG), first navigate to the menu item "Design-Time → SCG Extraction" (http://127.0.0.1:8090/design/scg-extraction.html). First, select all binary files (*.jar) belonging to the application. This process does not work solely on the source code files, because there are often dependencies to libraries and then the type resolution does not work. In addition, as with the instrumentation process, you can specify whether the mapping between source code elements and architectural model elements should be extracted before building the SCG. After clicking on "Extract SCG", the SCG should be successfully built and then be visible in the right half of the dialog.

System Extraction Process

In the following, the process of building the system model is explained step by step with pictures.

Step 1 - Open the appropriate tool

Select the menu item "Design-Time → System Extraction" (http://127.0.0.1:8090/design/system-extraction.html). The appearing screen should look like shown in the picture.

Step 1 - Initial form
Step 2 - Starting the process and select system interfaces

The next step is to launch the construction process. To do this, first click on the "Start Building Process" button at the bottom of the screen. Afterwards, the interfaces that should be provided by the system have to be selected. This is done in the right half of the tool in the view "Current repository model". Select an interface by double-clicking it. A selected interface can be recognized by a check mark. You can also deselect the interface by double-clicking it. If the running example is used, you should only select the interface "PrimeManager". After all interfaces have been selected, the actual building process starts by clicking on "Start Building System" at the bottom of the screen.

Pipeline Execution #back to top

In the following it is explained how the runtime features can be used. The first part explains how monitoring data is generated. Then the tool for observing the pipeline execution is presented. Finally, it is explained how the validation results can be accessed.

Run Instrumented Application

There are two ways to execute the instrumented running example:

  1. executing the class "dmodel.example.primes.PrimeStarter" in the previously generated instrumented project (see Instrumentation).
  2. executing the class "dmodel.example.primes.PrimeStarter" in the already included project "dmodel.example.primes.instrumented".

Observe Pipeline Execution

The execution of the pipeline can be observed by selecting the menu item "Runtime" in the sidebar (http://127.0.0.1:8090/runtime/index.html). This view shows the execution status for the most important transformations. It also provides direct access to the validation results.

Additional Features #back to top

There are a lot of other features, but in the following we want to talk about two of them in particular. On the one hand the dashboard, which summarizes the current status at a glance. On the other hand the integrated tool for managing the models.

Dashboard

The dashboard contains a lot of useful information if you want to investigate the state of the application. On the left side you can see the states of the application components. If there are problems with certain components, you can also see them there. On the right side, information about the pipeline executions is summarized. This particularly concerns performance characteristics (execution time, number of records processed). The dashboard can be viewed on the landing page (http://127.0.0.1:8090/index.html).

Models Summary

You can open the overview of the models via the menu item "Models Summary" (http://127.0.0.1:8090/models/summary.html). On this page you will find a tree view of all models in use. The elements within the models can be explored and the associated attributes and their values can be viewed.
